## 范围调整：MVP 暂不收 BGM 文件

实施到 Task 13（smoke）时用户做了关键决策：**MVP 不收 BGM 素材库**。

**理由（用户原话整理）**：
1. **纯 TTS 朗读已足够验证产品市场契合度**——爱宝故事的核心价值是"以孩子为主角的个性化叙事"，BGM 是体验加分项而非必需项
2. **BGM 筛选是 1-2 小时人类品味判断工作**——10 首素材需要符合 5 类风格 × 2 个变体的情绪曲线，AI 替代不了
3. **押在 critical path 上是错配**——代码部分是确定性工作可外包给 subagent，但素材筛选堵在前面会卡住整个 plan 上线
4. **等用户反馈想要音乐时再花那 1-2 小时**——届时零代码工作启用，今天投资的 dashboard 可观测性立刻就能告诉我们"启用前后差多少"

**今天的可观测信号**：
- `bgm_not_found_total{mood="adventure"}` 等 mood 标签的 counter 在生产中每次故事生成都 +1（100% degradation 率）
- 未来 BGM 上线后，这个比率会**逐步下降到接近 0**——dashboard 一眼可见
- `audio_mix_total{status="degraded"}` 同步 incrementing；启用后 `status="success"` 取代之

这是一个**"基础设施先于内容"**的设计选择——把可外包的代码工作做完，把不可外包的人类判断工作延后，让 critical path 永远是"人能立刻接力"的状态。详见下文工程教训第 3 条。

## 工程教训

### 1. 为指标投资永远值得 —— Plan 6b 的种子在 Plan 7 兑现

`BGMNotFoundTotal` counter 是 Plan 6b 在补 fail-open metrics 时顺手加的（当时还没 Plan 7 的概念）——只是为以后埋一个位。Plan 7 实施时没收 BGM 文件，这个 counter 立刻变成"100% degradation"的 dashboard 信号——**为指标投资永远不会浪费**，最坏情况是数字一直为 0。同 Plan 4 `llm_call_total` / Plan 5 `audio_ready_total` 的逻辑。

### 2. Composer interface 模式 —— handler 持小接口，service 持完整结构

worker handler 调用 `audio.Orchestrator`（持有 TTS / BGMRepo / Cache / Mixer 4 个 dep）时，**没有让 handler 直接持有 `*audio.Orchestrator`**——而是在 handler 包里定义一个只有一个方法的 tiny `Composer interface`：

```go
type Composer interface {
    Compose(ctx context.Context, req audio.ComposeRequest) (*audio.ComposeResponse, error)
}
```

production：`audio.Orchestrator` 隐式满足（Go 鸭子类型）。
test：写个 `stubComposer{...}` 就完成 handler 单测，不需要 mock 4 个底层依赖。

这是 [5.17 Composer/Handler 解耦](../knowledge/05-software-design.md#517-composer-handler-解耦) 的落地范例——ISP（Interface Segregation Principle）：客户端只依赖它实际用的最小接口，不依赖 service 完整 API。同 Plan 6 `bootstrap.ChildLookup` 接口。

### 3. 基础设施先于内容（推广原则）

Plan 7 选择把整条 BGM 管线**全部代码写完上线**但**不收一首素材**。可推广到任何"代码驱动 + 内容驱动"双轨任务：
- 代码部分是**确定性工作**——可以委托给 subagent 并行执行，可以单元测试，可以预测时间
- 内容部分（在本场景是 BGM 筛选）是**人类品味判断**——AI 替代不了，必须用户亲自坐下 1-2 小时

**错配陷阱**：把内容工作压在 critical path 上 → 代码做完了卡在最后一步等人选素材 → 全 plan 上不了线。
**正确做法**：代码先行，留预留接口和指标；内容延后填补，零代码改动启用。
