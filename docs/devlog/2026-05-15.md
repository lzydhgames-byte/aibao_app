## 架构落地

| 模块 | 实现 |
|---|---|
| **storylines 表** | id / child_id / title / status (active/completed/abandoned) / next_episode_hint / episode_count / last_episode_at；含 `stories.storyline_id` 外键 |
| **StorylineRepo** | Create / FindByID / ListActiveByChild / IncrementEpisode（单 UPDATE 原子递增） |
| **ChapterHookExtractor** | doubao-lite **顺势调用**：用 20 字内写下集预告。fail-open（空字符串），不阻塞主流程 |
| **StorylineContextBuilder** | 拼装 3 集 summary + previous hook + previous elements (top 5-8 角色/地点)。各子查询 fail-open |
| **system_prompt.tmpl** | 新增 `## 上一集剧情（连续剧第 N 集）` 段，位置在 IDENTITY 之后、MemorySummary 之前；措辞祈使 "请承接这些角色和场景写新一集" |
| **PostCheck 续集硬约束** | `RequireContinuity + PreviousElements` 字段；LLM 输出必须 `strings.Contains` 至少 1 个 PreviousElement，否则 `not_continuing` → fallback |
| **Orchestrator 主修改** | GenerateParams 加 `StartStoryline bool + StorylineID *int64`；start_storyline 路径预创 row、storyline_id 路径走 ctxBuilder；后扣 IncrementEpisode（带 chapter_hook）|
| **GET /heartbeat** | 时段感知问候（早/中/午/晚/夜深 5 段）+ 活跃 storyline 列表（限 5）。纯查询，无 cron |

## 已知小问题 / 未来优化

1. **`story_elements` 抽取偏稀** — Plan 4 启发式 extractor 通常只能识 1-2 个 place + 0 character。这次 ep1 只抽到 "森林"，没抽到 "小熊"。如果 PreviousElements 只有 1 个，PostCheck `not_continuing` 误判率会升高。**未来**：把 element extraction 升级成 LLM 顺势调用（复用 doubao-lite），或扩展启发式词表加常见儿童角色名。
2. **chapter_hook 输出风格偏文言** — "次日爱宝小宇又将在森林遇何事" 像章回小说。**未来**：在 chapter_hook prompt 加约束 "请用孩子能懂的口语，不要文言"。
3. **storyline status 永远 active** — 没有 Close()/Abandon() 接口。**未来**：暴露 PATCH /storylines/:id/close 让家长主动结束；或自动规则（30 天无新集 → abandoned）。
4. **storyline.title 永远空字符串** — Plan 8 没让 LLM 生成 storyline 标题。**未来**：第 1 集生成完时顺势调 doubao-lite 给 storyline 起个名（HEARTBEAT 卡片就能显示 "继续小恐龙的冒险"）。

## 工程教训沉淀

写进知识库 + MEMORY.md 的有以下两条，今天最值得复盘的是后者：

### 1. LLM "顺势调用"模式（知识库 11.11）

主调用做主任务，**顺势再调一次低成本模型** 做辅助提取——是 LLM 应用工程的"高 ROI"模式。
- Plan 6 落地：memory.Summarizer（主故事 → 30 字 summary）
- Plan 8 落地：chapter_hook.Extract（主故事 → 20 字预告）
- 共同特征：< 100 token 输出、用 doubao-lite（便宜 5-10 倍）、fail-open 不阻塞、产物结构化存进 DB 喂回下次主调用

### 2. 工程化"剧情连续感"：数据 + prompt + 校验三层叠加（知识库 5.18）

最重要的领悟：让 LLM **稳定**表现出连续剧质感**不能只靠 prompt**，要三层联动：
- **数据层**：storylines 表（一级实体）+ stories.storyline_id 外键约束 — 让"哪个是上一集"在多次请求间稳定关联
- **Prompt 层**：靠前位置的"## 上一集剧情"段 + 祈使措辞 "请承接这些角色和场景" — 给 LLM 显式承接信息
- **校验层**：PostCheck 引入 `RequireContinuity + PreviousElements` 硬约束 — 不承接的输出直接被拒、走 fallback

单做任一层都不够：单 prompt 是 Plan 6 状态（彩蛋率 0）、单数据是无指令的 LLM 自由发挥、单校验是"全拒绝但 LLM 不知道要承接什么"。

**Plan 6 → 6b → 8 的演进轨迹**就是"彩蛋质量的技术栈叠加"——从"什么都不做"到"位置×措辞×内容浓度"（Plan 6b）到三层联动（Plan 8）。
