### 5 项修复（按 commit 顺序）

#### Fix 1：BOOTSTRAP 润色不传 nickname → 编造名字（commit `bb50775`）

- **症状**：Plan 6 smoke 中 children.profile.description 出现"小明是个温柔又勇敢的孩子"——孩子实际叫"小宇"，LLM 凭空编造。
- **根因**：`bootstrap.renderDescription` 只把 7 个 BOOTSTRAP 答案传进 LLM prompt，**没传 child.Nickname**。
- **修复**：`bootstrap.Service` 引入 `ChildLookup` 接口（依赖倒置），调用 LLM 前先 `childLookup.Get(ctx, childID)` 拿 Nickname → 进 prompt 模板。同时埋点 `llm_fail_fallback_total{provider,model,reason="bootstrap_polish"}`。
- **smoke**：description = `"小宇是个勇敢又好奇的孩子，他最喜欢的角色是奥特曼..."`（不再有编造名字）✅

#### Fix 2：PostCheck 严格度按 duration 缩放（commit `80cc731`）

- **症状**：5 分钟短故事 nickname 出现 1-2 次很常见，但 `minProtagonistOccurrences=3` 把它们打成 `child_not_protagonist` → 全部走 fallback 模板。
- **根因**：Plan 3 把阈值写死 = 3，没考虑时长。
- **修复**：新增 `minProtagonistFor(duration)`：`5min → 2 / 10min → 3 / 15min → 4`。`PostCheckInput.Duration` 字段加上，orchestrator 把 `p.Duration` 透传。
- **smoke**：5 min story_id=23，小宇出现 14 次（≥2 阈值），used_fallback=false ✅

#### Fix 3：memory 软提示位置 + 措辞双升级（commit `fd6c782`）

- **症状**（Plan 6 known issue 3）：memory context 注入了 675 字到 system prompt 末尾、措辞"可以自然回调"——LLM 完全无视。
- **修复**：
  1. **位置**：从 system prompt 末尾移到 IDENTITY 段之后、8 条强约束之前
  2. **措辞**："可以自然回调" → "请尝试借用以下记忆里的角色或场景，让孩子感受到熟悉的延续"
  3. **保持**：30 字精炼摘要不变（避免长文稀释）
- **smoke**：Story 23 文本 `"我们遇到了上次一起玩耍的小精灵"`——**真回调 Story 22 的小精灵** ✅
- **教训**：单独改一项可能没效果，**位置 + 措辞 + 内容浓度三维同时调整**才达成质变。详见知识库 11.10 末尾追加段。

#### Fix 4：Nickname UTF-8 入口校验（commit `8a4d158`）

- **症状**（Plan 6 遗留）：Plan 4 时 Git Bash + GBK locale 让 id=3 nickname 进 DB 时是乱码 `С��`。
- **直觉写法行不通**：先 `c.ShouldBindJSON(&req)` 再 `utf8.ValidString(req.Nickname)` **永远返 true**——因为 `encoding/json` 在 Unmarshal 时静默把非法字节替换成 U+FFFD（合法 UTF-8 字符）。
- **正确姿势**：读 raw body → `utf8.Valid(raw)` → restore body → ShouldBindJSON。
- **修复**：POST/PATCH `/children` 入口都加上这个三步序。返 `400 invalid_utf8`。
- **smoke**：POST /children body 含字节 `0xc8 0xed 0xa1 0xa1` → HTTP 400 ✅
- **教训沉淀**：知识库新增 **6.13 入口校验必须在 JSON unmarshal 之前**。

#### Fix 5：llm_fail_fallback_total counter（commits `9e6765f` / `72f7007` / `b5d1567` / `bb50775`）

- **症状**（Plan 6 教训沉淀）：Plan 4 时 intent endpoint 绑了文生图 Seedream 模型，意图分类调用全 404 → fail-open 兜底 IntentSafe → 业务无感 → Plan 4/5 沉默两个 plan，Plan 6 才暴露。
- **修复**：定义 `llm_fail_fallback_total{provider,model,reason}` CounterVec，在 3 个 fail-open 决策点埋点：
  - `intent_llm.Classify`：reason=`upstream_error` / `parse_error`
  - `memory.Summarizer`：reason=`upstream_error` / `empty_output`
  - `bootstrap.renderDescription`：reason=`upstream_error` / `template_render_error`
- **Prometheus 行为说明**：CounterVec 在第一次 `.Inc()` 之前**不会出现在 /metrics 输出**——这是设计哲学（label 组合无穷可能，不预先全枚举）。Plan 6b smoke 因没有失败，所以 counter 未出现 = 正常。
- **smoke**：counter 已注册，单测验证 inc 路径；smoke 无失败 → 0 行 = 符合 Prometheus 约定 ✅
- **教训沉淀**：知识库 9.14 末尾补一段"counter 不出现 ≠ 未注册"。
