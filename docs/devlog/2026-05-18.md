### 接力卡 #3：多样性 / 同质化抽样验证

#### 意外发现：同 prompt → 同故事（commit `3231d8f`）

跑抽样实验前，用户偶然发现："迪士尼乐园的一天" 输入两次产生**字面相同**的故事。

🎓 **根因**：identical system+user prompt → Doubao prompt-prefix 缓存命中 → 同输入同输出。

**修法**：在 user role 末尾追加三件套：
```
[本次创作随机灵感] <SceneSeed>
[多样性要求] 即使主题或需求和之前讲过的故事相同，也请写一个完全不同的情节、不同的转折、不同的角色配置。
[本次会话 ID] <16-hex nonce>
```

system prompt 不动（保留稳定前缀的缓存收益），只在 user role 末尾打破——nonce 是纯破缓存（LLM 视为噪声但 bytes 不同），mandate 是显式多样性指令，seed 是 LLM 实际能用的情节抓手。

复测：同 prompt 两次 → 完全不同的故事。

#### 长度保护阈值收紧 70% → 85%（commit `e374f56`）

8min 故事 LLM 写 1500-1611 字时，旧 70% 阈值 (1612) 让它"刚好压线放行"，实际音频只有 4:40。新阈值 85% (1958) 把这类"几乎够但用户感知短"的输出都送进重写。

#### SceneSeed 审计埋点（commit `2d74ad3`）

`BuildOutput` 新增 `SceneSeed` 字段，orchestrator 打 `prompt.scene_seed` info 日志，方便事后审计 80 个种子的轮转分布。

#### 多样性抽样实验：15 个故事

**实验设计**：
- 第 1 轮 10 个故事，10 个不同主题 prompt
- 第 2 轮 5 个故事，5 个含教育主题的 prompt（专门测 negative_values warn-only）

**结果**：

| 指标 | 第 1 轮 (10) | 第 2 轮 (5) | 合计 |
|---|---|---|---|
| 通过（非 fallback） | 9/10 (90%) | 4/5（PreCheck 拦截 1） | 13/15 (87%) |
| fallback | 1/10 (「自私」) | 0/5 | 1/15 (7%) |
| PreCheck 拦截 | 0 | 1 (「嘲笑别人」用户输入端) | 1/15 |
| SceneSeed 唯一性 | 10/10 | 5/5 | **15/15 全唯一** |
| 平均生成时间 | ~62s | ~63s | 一致 |

**story 63 那个 fallback** 是 negative_values warn-only 改之前发生的，改完之后第 2 轮全过 → 验证 warn-only 修复正确。

### Android Studio AVD 替代 MuMu（commit `8d4b6f2`）

MuMu 模拟器今天暴露了多个不稳定问题：

1. **方案 A（保守）**：PreCheck 同样接入 category 信息，negative_values 类放宽到 warn-only 或直接跳过

3. **方案 C（懒）**：让 PostCheck 兜底，PreCheck 全放（不推荐——浪费 LLM 调用）

倾向 A：跟 PostCheck 改造对称。

## 关键教训

1. **过滤的代价是 UX，而非"安全"**——红线词清理 5 轮才稳定，每一轮删的都是上一轮我没意识到的误伤场景。"宁滥勿缺"的内容过滤直接把 30% 故事打成 1 分钟罐头，比"漏放一些悬念词"严重得多。
2. **PreCheck/PostCheck 应该对称设计**——今天 PostCheck 改了 horror+negative_values warn-only，但 PreCheck 没动，导致用户端拦截依然过严。
3. **prompt cache 是默认开启的**——任何 LLM 后端如果允许重复 prompt，必须主动设计 cache buster。
4. **多样性 = 输入熵**——SceneSeed 80 池 + nonce + mandate 三件套联动，每个故事开头都不一样。
5. **token ≠ 汉字**：`output_tokens` 大约 = 汉字 × 1.3-1.5，写代码判断字数时一定要用 `CountCJKRunes(text)` 而不是 token 数。
6. **AVD > MuMu**：稳定性是工具链的硬通货。
