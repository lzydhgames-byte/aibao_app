### 接力卡 #1：PreCheck 反义教育 prompt 处理（commit `f26257a`）

**起因**：昨天发现 "不要嘲笑别人" 这种**反义教育 prompt**被 PreCheck 直接 400 拒绝。PostCheck 已经把 horror/negative_values 改了 warn-only，但 PreCheck 没动 → 用户根本走不到 PostCheck。

**修法**：让 PreCheck 跟 PostCheck **对称设计**——同一份分级表，soft 类记 warn 而非 reject。

实现要点：
1. **RuleSet 新增 `WordToCategory`**：rules.yaml 加载时一次性建反查索引。PostChecker 也改用它（之前自己算副本）。
2. **PreCheckOutput 新增 `MatchedCategory` + `SoftWarnings`**：保留软命中信息给调用方。
3. **`KeywordMatcher.FindAll`** 新方法：必须扫所有匹配。否则"软 + 硬"混合 prompt 可能因为 soft 先命中就放行了真正的 hard。
4. **PreChecker.Check** 重写：扫所有红线匹配，soft 入 warnings，第一个 hard 立即 reject。
5. **orchestrator** 打 `story.precheck.soft_warn` 日志，便于审计。

**验证**（5 个教育 prompt）：

### 字数系数 320 → 280 + 区间 ±10% → ±5%（commit `dde3c39`）

**起因**：15 个 5min 故事平均超时 +14%，最坏 +33%。LLM 把上限当目标。

**修法**：
- `expectedRunesForDuration` 系数 320 → 280
- 区间 ±10% → ±5%
- 5min 中位 1600 → 1400，区间 [1330, 1470]
- 长度保护 85% 阈值跟着降到 1190（之前 1224，差异微）

**新增成本 metric**：
- `tts_chars_total{duration_min}`：累计送 TTS 的汉字数（账单基础）
- `tts_chars_excess_total{duration_min}`：超出区间上限的汉字数（纯浪费）

实施改动：`audio.ComposeRequest` 新增 `DurationMin`，audio orchestrator 在 TTS 调用前埋。

## 关键教训

1. **Pre/Post 必须对称设计**——任何分级宽容做在下游而上游一刀切，等于没做。
2. **数据驱动调参**：5min 调对完全靠 15 故事样本反推（实际速率 250-280 字/分钟，不是预设的 300）。
3. **LLM 训练倾向是硬墙**：prompt 工程到一定深度就撞 Doubao 的"故事完整收尾"训练偏好——加结构、加字数、加重写都越来越无效。8min 用户必须降低期待，或换模型。
4. **lint 工具是技术债的早期偿还**：今天加 rules-lint 才半小时，避免未来一次手抖造成 silent bug。
5. **Token ≠ 字数 ≠ 朗读时长**：output_tokens 1500 ≈ 汉字 1200 ≈ TTS 时长 4:30——三个尺度，做长度判断时永远算汉字。
