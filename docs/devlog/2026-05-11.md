## Issue 与教训

### Issue 1：COS bucket name 双 APPID 陷阱（commit 2f8f0f3）
- **现象**：第一次跑 COS 上传 → `NoSuchBucket: The specified bucket does not exist`。
- **根因**：腾讯云 COS 控制台显示的 bucket 名是**完整名**形如 `aibao-audio-1234567890`（短名 + APPID），运维者（也就是用户本人）把这个完整名贴进 `COS_BUCKET` 配置。但 cos-go-sdk-v5 期望传入**短名**——它内部会自动拼 `-<APPID>`。结果代码访问的 bucket 变成 `aibao-audio-1234567890-1234567890`——404。
- **修复**：cos.go 加 detect-and-skip 守卫：
  ```go
  if !strings.HasSuffix(cfg.Bucket, "-"+cfg.AppID) {
      cfg.Bucket = cfg.Bucket + "-" + cfg.AppID
  }
  ```
- **影响**：本地 smoke 浪费 ~30min 定位。生产同样会踩——只要运维有人贴完整名就中。代码层守卫是必须的。
- **教训沉淀**：已加入知识库（云服务标识符的"完整名 vs 短名"陷阱）。

### 经验：infra 连通性预测试早做早省时
Plan 4 教训沉淀，Plan 5 接 COS 和 Minimax 时分别先用一次性 `smoke-cos/main.go` / `smoke-tts/main.go`（脱离项目代码、裸 SDK 调用）验证凭证 + bucket + voice_id。提前发现了 region 不匹配 + 子账号策略未绑定 + 上面 Issue 1 的 bucket 双拼问题——如果这些在应用层 smoke 才暴露，会被业务逻辑包装的错误信息淹没，定位至少 ×3 时间。**已加入知识库 6.12**。

### 经验：异步路径不是"为了快"，是"为了感知快"
TTS + COS 上传**实测**只用 6 秒。同步路径理论上总等待会从 16s 涨到 22s——绝对值只多 6 秒，但用户体验区别巨大：异步路径用户在 16s 拿到文本可以**立刻开始阅读**，等他读完第一段音频也好了，感知延迟接近零；同步路径用户必须呆等 22 秒一片空白。**已加入知识库 5.15**。
