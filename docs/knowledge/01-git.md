# Git 知识

## 1.1 Git
**版本控制系统**——记录代码每次变化，能随时回到任何历史版本。  
**类比**：自动版"另存为 v1 / v2 / v3 final"，但永远找得回。  
**为什么需要**：项目越大改动越多，没有版本控制等于"改坏了再也救不回来"。Git 把"存档"做到极致：每次提交都有时间戳和说明，可比对、可回滚。

## 1.2 `git init` / Git 仓库
在当前文件夹创建 `.git/` 隐藏目录，让 Git 开始追踪这里的变化。每个项目只做一次。  
**为什么需要**：Git 默认对随便一个文件夹**什么都不做**。`git init` 等于告诉 Git "请开始记账"，之后所有 commit 才有地方存。

## 1.3 全局身份（user.name / user.email）
每个 commit 都要"签名"。全局身份是默认作者，每台电脑配一次。  
不必真实——只是元数据；推 GitHub 时才需要邮箱匹配。  
**为什么需要**：多人协作时要知道"这行代码是谁写的、出问题找谁"。即使单人开发，git log 里不署名也很奇怪。

## 1.4 分支（branch）
Git 里的"平行宇宙"，多条开发线互不干扰。  
默认主分支以前叫 `master`，业界已改名 `main`。MVP 阶段我们只用 `main`。  
**为什么需要**：想试一个大改动又不敢动主线？开个分支随便玩，搞砸了删掉分支主线毫发无伤；满意了再合并回来。多人协作时每人一个分支也避免互相覆盖。

## 1.5 工作流：工作区 → 暂存区 → 仓库
```
改文件 → git add <文件>  → git commit -m "..."
         （选中要打包的）   （拍快照）
```
**为什么需要分两步**：一次改了 5 个文件，但本次只想 commit 其中 2 个相关的。`git add` 让你**精挑细选**进入快照的内容，避免"被迫一次提交全部"。

## 1.6 commit message + Conventional Commits
commit 时附的一句话说明。业界惯例用前缀分类：

| 前缀 | 含义 |
|---|---|
| `feat` | 新功能 |
| `fix` | 修 bug |
| `refactor` | 重构（不改外部行为） |
| `docs` | 文档 |
| `chore` | 杂事（配置、工具） |
| `test` | 加 / 改测试 |

好的 message 写给"3 个月后的自己"——讲清楚**为什么**改，不只是改了什么。  
**为什么需要前缀**：3 个月后翻 git log，前缀让你一眼分清"功能改动"和"杂事"；自动化工具（生成 changelog、决定版本号）也靠它。

## 1.7 `.gitignore`
告诉 Git "这些文件假装看不见"——编译产物、日志、IDE 配置、**密钥**。  
**为什么需要**：没它的话 `git add .` 会把无关文件（编辑器临时文件、密码）一股脑入库，污染仓库还可能泄密。  
**铁律**：密钥**绝不能 commit**。Git 历史是永久的，删了下个 commit 也救不回来——因为 `.git/objects/` 里那个版本永远在。

## 1.8 `.gitkeep` 占位
Git 不追踪空目录。在空目录里放一个空文件 `.gitkeep`，Git 就会记录这目录。  
不是 Git 语法，只是社区惯例的名字。真实文件落进来后就该删掉它。  
**为什么需要**：你规划好了目录结构（`internal/api/`、`internal/service/`），但还没写代码——`git clone` 下来发现这些目录都没了，因为 Git 把空目录当不存在。`.gitkeep` 让 Git "看到"它们。

## 1.9 commit 哈希（SHA）
每个 commit 的"身份证号"，如 `8a17a07`。不可伪造——改任何字节哈希就变。  
通常用前 7 位即可：`git show 8a17a07`、`git diff 8a17a07 HEAD`。  
**为什么需要**：分支名、tag 都可能改名/重指；commit 哈希是**永久不变的精确坐标**——出 bug 时"这条 bug 是从 8a17a07 之后出现的"是绝对可信的描述。

## 1.10 LF vs CRLF
Unix 用 `\n`（LF），Windows 用 `\r\n`（CRLF）。Git 在 Windows 上自动转换。  
看到 `LF will be replaced by CRLF` **是提示不是错误**。  
**为什么会这样**：历史包袱——三大操作系统对"换行"达不成共识。Git 的策略是"仓库里永远 LF（统一），本机 CRLF（Windows 友好）"，自动转换让你不用操心。
