# Docker 与容器

## 4.1 Docker
把软件 + 它的所有运行环境打包成"盒子"，运行时不污染你的系统。  
**类比**：搬一套完整西藏帐篷到你家——用完撤走，地面没动过。  
**为什么需要**：传统装 PG 要下载、配置、注册系统服务、改配置——卸载残留一堆。Docker 让"装个软件"变成"启个容器"，要删一秒删干净。开发机干净、CI/CD 环境一致。

## 4.2 容器 vs 虚拟机
- **虚拟机**：模拟整个 OS，几 GB，启动慢，隔离极强
- **容器**：共用宿主机内核，只打包应用，几十 MB，启动快

**类比**：虚拟机=独立楼；容器=公寓里租一间房。  
**为什么各有所用**：要跑完全不同的 OS（Windows 上跑 Linux 内核功能）→ 用 VM；只是隔离应用环境 → 容器更划算。开发环境 99% 用容器够了。

## 4.3 镜像 vs 容器
- **镜像**：只读模板（如 `postgres:16`）
- **容器**：基于镜像跑起来的实例（如 `aibao-postgres-dev`）

**类比**：镜像=Word 模板；容器=基于模板新建并填了内容的文档。一个镜像可建无数容器。  
**为什么这样设计**：镜像只读+分层缓存让传输/存储极省（多个容器共享底层只读层）；容器可读写让运行时数据有地方放。一个镜像启 10 个容器，磁盘只多一份镜像 + 10 份容器各自的写入差异。

## 4.4 docker-compose
用一个 YAML 描述"我要启哪些容器、怎么互联"，一行命令全部启动：
```bash
docker compose up -d        # 启动
docker compose down         # 停止（保留数据）
docker compose down -v      # 停止 + 删数据卷（彻底清空）
```
**为什么需要**：开发环境通常要 PG + Redis + 你自己的服务一起跑——一个一个 `docker run` 累且容易写错参数。compose 把"启动姿势"固化在 yaml 里，新人 `docker compose up` 立刻有完整开发环境。

## 4.5 端口映射 `127.0.0.1:5432:5432`（安全）
绑定 `127.0.0.1` 而不是默认的 `0.0.0.0` —— **只允许本机访问**，不暴露给局域网/公网。  
如果写成 `5432:5432`，同 WiFi 下别人能直连你的数据库。  
**为什么需要小心**：开发数据库密码通常很弱（我们用 `aibao/aibao`）。绑 `0.0.0.0` 等于把后门开给整个 WiFi——咖啡馆里随便有人扫端口就连上了。`127.0.0.1` 限制只本机能连，安全多了。

## 4.6 数据卷（volumes）
容器是临时的（删了就没），卷是持久的——把数据存外面：
```yaml
volumes:
  - aibao_pg_data:/var/lib/postgresql/data
```
容器删了数据还在；`down -v` 才删卷。  
**为什么需要分离**：如果数据存容器内部，容器删了数据全没——开发时这是灾难。卷把"数据"和"运行时"分开：容器随时可销毁重建，数据持续存在。生产环境同理（容器无状态 + 数据外置存储）。

## 4.7 healthcheck
让 Docker 定期"问候"容器（如 `pg_isready`），判断**服务是否真能用**——不只是进程活着。  
其他依赖此服务的容器可以等到 healthy 才启动。  
**为什么需要**：进程在跑 ≠ 服务正常。比如 PG 进程启动后内部初始化还没完，连接会失败。靠 healthcheck 实际"问候一下"才能判断。依赖此服务的容器可以等 healthy 才启——避免"DB 还在启动你的 app 已经连过去报错"。

## 4.8 `docker exec`
在已运行的容器里执行命令——借用容器内的客户端：
```bash
docker exec aibao-postgres-dev pg_isready    # 不用本机装 psql
docker exec aibao-redis-dev redis-cli ping
```
**为什么方便**：你本机不用装 psql / redis-cli——容器内自带这些客户端，直接借用。开发机更干净，工具版本和容器内 server 也保证一致。
