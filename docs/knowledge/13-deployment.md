# 部署上线

> Plan 10 把爱宝从"本机能跑"变成"互联网上朋友能用"。这个文件汇总把一个 Go 后端 + Flutter App 部署到云服务器的完整知识链。

## 13.1 部署（Deployment）

**一句话**：把代码从开发机搬到一台 7×24 小时开机、有公网 IP、别人能访问的服务器上跑起来。

**生活类比**：开发是在自家厨房试做菜谱，部署是把菜端到餐厅后厨、挂上招牌、开门迎客。试做时可以随便尝、随便改；开门后客人随时进来，后厨不能停。

**部署 ≠ 写代码**。部署是一整套独立技能：选服务器、装运行环境、配网络、上证书、设开机自启、做监控。代码 100% 写完，部署可能还要花一两天。

**何时引入**：Plan 10。

## 13.2 云服务器：轻量应用服务器 vs CVM

**一句话**：腾讯云的"轻量应用服务器"是简化版云主机，固定套餐价、配置好网络，适合中小项目；CVM 是完整版，按需精细配置。

**爱宝选型**：香港轻量 2C4G / 70GB SSD / 30Mbps。2 核 CPU、4GB 内存，对单体 Go 服务 + ~1000 测试用户够用。

🎓 **为什么选香港**：跳过中国大陆 ICP 备案（备案要 2-3 周）。代价是大陆访问稍慢、且大陆正式上线前必须迁回境内（儿童数据合规要求）。MVP 期香港是"快速上线"的合理取舍。

**何时引入**：Plan 10 Day 1。

## 13.3 服务器当"牲口"不当"宠物"

**一句话**：服务器应该是可随时重建的、配置全部脚本化的"牲口"，而不是手工调教出来、坏了没人会修的"宠物"。

**生活类比**：宠物有名字、生病要心疼、死了无法替代；牲口编号管理、坏了直接换一头。服务器要当牲口——所有配置（装什么、怎么配）都写进脚本和文档，这台挂了能在 30 分钟内拿一台新的照着脚本重建。

**为什么需要**：手工 SSH 进去敲命令配出来的服务器，三个月后没人记得当初装了什么、改过哪个配置文件。一旦坏了或要扩容，就是灾难。

**爱宝实践**：启动脚本、`/etc/aibao/env`、systemd unit、nginx 配置都有记录；devlog 里写明每一步。

🎓 **反面教训**（Plan 10 Day 1 实战）：服务器**不是干净的**——前一个项目留下了 Docker 容器、系统用户、跑着的服务。如果不先 `docker ps -a` / `ls /opt` / `netstat` 探一遍就动手，新项目会跟前任的容器/端口/规则打架。**接手任何服务器先做一次"体检"**。

**何时引入**：Plan 10。

## 13.4 systemd：让程序开机自启 + 崩溃自愈

**一句话**：systemd 是 Linux 的"服务管家"，负责开机自动拉起你的程序、程序崩了自动重启、统一收集日志。

**生活类比**：systemd 像餐厅经理。后厨厨师（你的程序）下班了它负责第二天开门时叫醒；厨师中途晕倒它立刻顶一个上去；厨师说的每句话它都记在本子上（日志）。

**核心概念——service unit 文件**：一个 `.service` 文本文件描述"怎么管这个程序"。爱宝的 `/etc/systemd/system/aibao.service` 关键字段：
- `ExecStart` — 启动命令（跑哪个二进制）
- `EnvironmentFile=/etc/aibao/env` — 从哪个文件读环境变量（密钥）
- `Restart=on-failure` — 程序异常退出时自动重启

🎓 **`Restart=on-failure` 解决的真实痛点**：Plan 9d 发现 PostgreSQL 容器中途挂掉后，Go 后端优雅关停了但不会自动重连。有了 systemd 的自动重启，后端进程挂掉能被自动拉起——这是生产环境的"兜底保险"。

**常用命令**：
- `systemctl start/stop/restart aibao` — 启停/重启
- `systemctl enable aibao` — 设为开机自启
- `systemctl status aibao` — 看运行状态
- `systemctl is-active aibao` — 只回 active/inactive
- `journalctl -u aibao -f` — 实时看日志（`-f` = follow，像 tail -f）

**何时引入**：Plan 10 Day 1。

## 13.5 Nginx 反向代理

**一句话**：Nginx 站在你的程序前面，对外接所有请求，再按规则转给后面真正干活的程序——这叫"反向代理"。

**生活类比**：Nginx 是公司前台。访客（用户请求）都先到前台，前台根据"找谁"把人引到对应办公室（后端程序 / 静态文件 / 落地页）。访客不需要知道每个部门在几楼。

🎓 **为什么叫"反向"**：普通代理（正向代理）代表**客户端**出门访问外网（如翻墙代理）；反向代理代表**服务器**对外接客。方向相反，所以叫反向。

**爱宝的 Nginx 路由表**：
```
listen 80 / 443  →  https://aibao.dhgames.com
/api/*                     → 转发到 127.0.0.1:8080（Go 后端）
/download/app-release.apk  → 直接吐静态文件（APK）
/                          → 落地页 HTML
```

**为什么需要反向代理**（不直接让 Go 程序对外）：
1. **统一入口** — 一个域名后面可以挂多个服务
2. **HTTPS 卸载** — 证书配在 Nginx 一处，后端程序不用管加密
3. **静态文件** — APK / 图片 / 落地页让 Nginx 直接吐，比 Go 程序快
4. **安全** — 后端只监听 `127.0.0.1`，外网碰不到，只能经 Nginx

**配套坑**：Nginx 默认不认识 `.apk` 文件类型，浏览器会把它当网页打开而不是下载。要加 MIME 类型 `application/vnd.android.package-archive`。

**何时引入**：Plan 10 Day 1。

## 13.6 HTTPS + Let's Encrypt + 证书自动续期

**一句话**：HTTPS = 加密的 HTTP；要开 HTTPS 需要一张"证书"；Let's Encrypt 免费签发证书，certbot 工具自动申请并定期续期。

**生活类比**：HTTPS 证书像营业执照——挂在门口证明"这家店是正经注册过的，不是李鬼"。浏览器看到有效证书才显示小锁图标；没有就警告"不安全"。

**关键概念**：
- **证书有有效期**。Let's Encrypt 证书只有 90 天（故意短，逼你自动化）
- **certbot** — 自动申请证书的工具，还会装一个定时任务自动续期
- 爱宝证书：`/etc/letsencrypt/live/aibao.dhgames.com/fullchain.pem`，2026-08-16 过期，certbot 已设自动续期

🎓 **为什么证书故意只给 90 天**：短有效期逼所有人把续期自动化。如果给 1 年，大家会手工续、然后忘记、某天网站突然全红。90 天 + 强制自动化 = 整个互联网更安全。

**为什么必须 HTTPS**：① 不加密的话，用户密码、孩子数据在网络上裸奔，任何中间节点可偷看 ② 现代浏览器对 HTTP 站点显示"不安全"警告，吓跑用户 ③ Android App 默认禁止明文 HTTP 请求。

**何时引入**：Plan 10 Day 1。

## 13.7 子域名跳过备案

**一句话**：在一个已备案主域名下开子域名（`aibao.dhgames.com`），通常不用单独再备案。

**爱宝实践**：用户公司已有备案主域名 `dhgames.com`，开子域名 `aibao` 给爱宝用，DNS 加一条 A 记录指向服务器 IP 即可。

🎓 **DNS A 记录**：DNS 是"互联网电话簿"，把域名翻译成 IP。A 记录就是一条"`aibao.dhgames.com` → `43.128.30.183`"的对应条目。用户改完后全球 DNS 服务器逐步同步（几分钟到几小时）。

**何时引入**：Plan 10 Day 1。

## 13.8 编译时配置注入（--dart-define / String.fromEnvironment）

**一句话**：Flutter 用 `--dart-define=KEY=VALUE` 在打包时把配置（如后端地址）"烧"进 App，代码里用 `String.fromEnvironment('KEY')` 读。

**爱宝实践**：
```dart
String baseUrl = const String.fromEnvironment(
  'API_BASE', defaultValue: 'http://127.0.0.1:8080',
);
```
```bash
flutter build apk --release --dart-define=API_BASE=https://aibao.dhgames.com
```
开发时不传 → 用默认本地地址；打 release 包时传生产地址。

🎓 **为什么不直接把生产地址写死在代码里**：写死了，开发调试时每次都要手动改回本地地址、提交前再改回去，极易出错。编译时注入 = "同一份代码，打包时决定连哪"。这是 12-Factor App "配置外置"原则在客户端的体现。

**何时引入**：Plan 10 Day 1。

## 13.9 release 构建：R8 minify 与 proguard keep 规则

**一句话**：Android release 包会用 R8 工具"瘦身"——删掉没被调用的代码。但有些代码是靠"反射"在运行时才调用的，R8 看不见会误删，需要用 proguard 规则手动"保护"。

**生活类比**：R8 像搬家时扔东西——"这半年没碰过的就扔"。但有些东西（应急药箱）虽然平时不碰，关键时刻要命。proguard keep 规则就是贴在这些箱子上的"别扔"标签。

🎓 **反射（reflection）是什么**：正常调用代码是写死的 `a.doSomething()`，编译器看得见。反射是运行时用字符串"按名字找代码来调"，编译器/R8 静态分析时看不到这条调用线 → 以为这段代码没人用 → 删掉 → 运行时崩溃。

**爱宝实践**：`just_audio` 底层的 ExoPlayer 大量用反射。`proguard-rules.pro` 里 `-keep class com.google.android.exoplayer2.** { *; }` 等规则保住这些类。Plan 9-A 时踩过坑——不加规则，release 包音频"准备好但一播放就报错"。

**为什么 release 要 minify**：debug 包不瘦身方便调试；release 包瘦身能显著减小体积（APK 更小、下载更快）。代价就是要正确写 keep 规则。

**何时引入**：proguard 规则 Plan 9-A 首次踩坑，Plan 10 打 release 包正式启用。

## 13.10 对象存储与计算资源同地域原则

**一句话**：你的服务器在哪个地域，对象存储桶（COS / S3）就尽量也开在哪个地域——跨地域不只是慢，大文件传输还可能直接卡死。

🎓 **爱宝血泪实战**（Plan 10 Day 1→Day 2）：服务器在香港，COS 桶最初在上海。小文件上传 300ms 秒过，但 5MB 音频文件 PUT 到上海桶，**在 TLS 握手阶段卡死 20 秒超时**。两次改后端代码（自定义 transport、裸 net/http）都没用——这不是代码层 bug，是香港→上海这条跨地域链路对大流量 HTTPS PUT 有网络层处理（MTU / 防火墙 / TCP window 协商问题）。

**修法**：把 COS 桶换到 ap-hongkong（和服务器同地域）。换完 TLS 卡死立刻消失。

**为什么需要**：跨地域链路对小请求没事，会让人误以为"配置没问题"。但音频、视频、大图这类大文件一上传就暴露问题，且症状（TLS 卡死）看起来像代码 bug，极难定位。**部署规划阶段就该让存储 region = 计算 region**，省掉一整天 debug。

**何时引入**：Plan 10 Day 2。

## 13.11 多层故障：剥洋葱式诊断

**一句话**：一个表面故障背后可能藏着多层独立问题，每修对一层才会暴露下一层——别指望一次改完，按"改一刀 → 看上游的原始回复 → 再改一刀"的节奏走。

🎓 **爱宝实战**（Plan 10 Day 2 的"三层洋葱"）COS 上传故障：
1. **网络层** — 跨地域 TLS 卡死 → 换同地域桶 → 修复后冒出下一层
2. **权限层** — 新桶 403 AccessDenied → 换对的密钥 → 修复后冒出下一层
3. **操作层** — 旧密钥 SecretKey 不可查看 → 新建密钥 → 全部打通

每一层在上一层没修好前根本看不见——网络不通时，永远走不到权限校验那一步。

**诊断方法**：
- 每修一步，**看服务的原始错误回复**（COS 返回的 XML、HTTP 状态码），不要凭感觉猜
- 错误码变了 = 你前进了一层（从 TLS timeout → 403，说明网络层已通）
- 把每层的"症状 → 假设 → 验证 → 修复"记下来（devlog 就是干这个的）

**为什么需要**：新手遇到"还是不行"容易怀疑"是不是上一步白改了"而回退。其实错误**变了**就是进展。识别"错误码变化 = 层数推进"能避免反复横跳。

**何时引入**：Plan 10 Day 2。
