# 软件设计原则与模式

## 5.1 关注点分离（SoC）
每段代码只做一件事。**类比**：餐厅切菜的、炒菜的、收银的各管一摊。  
项目体现：三层架构、viper（"从哪读"）+ mapstructure（"怎么填"）。  
**为什么需要**：一个函数同时管"读 yaml""填 struct""校验值""下载 BGM"——任何一处需求变化都要改这个巨型函数，越改越烂。分开后每段代码独立改、独立测，团队多人开发也不会撞车。

## 5.2 依赖倒置（DIP）
高层依赖**抽象接口**，不依赖**具体实现**。  
**类比**：插座（抽象）+ 灯泡（具体）——换灯泡随便换，墙不动。  
项目体现：LLM/TTS/SMS/Storage 全走 Gateway 抽象层；换豆包→Claude 业务零改动。  
**为什么需要**：业务代码如果直接 `import 豆包SDK`，将来换 LLM 提供商要改满地都是。让业务依赖 `gateway/llm.Client` 接口，换提供商只新增一个实现文件，业务代码零改动。也方便测试——单测时塞一个假实现进去。

## 5.3 YAGNI（You Aren't Gonna Need It）
没有真实需要时，**不要**提前引入复杂度。  
**类比**：刚搬家就买"以备不时之需"的备用沙发——客厅塞不下还都没用过。  
项目体现：MVP 不上 Docker、不拆微服务、不上 Kafka、Prometheus 暂不部署。  
**为什么需要克制**：每多一个组件就多一个故障点、多一份维护成本、多一份学习负担。"为未来准备"通常猜错——真到那天需求变了，前期工作要么没用要么过时。等到痛了再加，永远不晚。

## 5.4 12-Factor App（重点：配置外置）
**铁律**：代码里**绝不**写死配置。配置走环境变量。  
好处：同一份代码可跑 dev/staging/prod；密钥永不上 git；上线只改 env。  
项目体现：yaml + `AIBAO_*` env 覆盖；`config.prod.yaml` 在 `.gitignore`。  
**为什么需要**：硬编码配置 = 每个环境一份代码 = 部署灾难。把配置外置后，**同一个二进制可以跑在任何环境**——dev / staging / prod 行为不同只看注入的 env。密钥永不入 git 也是这个原则的副产品。

## 5.5 Pets vs Cattle
现代运维：服务器当**牲口**——挂了换新的，10 分钟拉起来。不当宠物精心抢救。  
做到这点的关键：配置外置、文件存对象存储、schema 走迁移工具、备份脚本入 git。  
**为什么需要**：把服务器当宠物意味着所有运维知识在管理员脑子里——他离职项目就废。当牲口意味着一切自动化（启动脚本、备份、监控都在 git 里），新人 10 分钟从零拉起一个完整环境，挂了也不怕。

## 5.6 幂等（Idempotent）
执行 N 次和 1 次结果完全一样。  
✅ `UPDATE users SET status='active' WHERE id=42` —— 跑 100 次都是 active  
❌ `UPDATE users SET balance = balance + 100 WHERE id=42` —— 跑 100 次扣 1 万 vs 100 块  
**类比**：电梯按钮——按一次和按十次都是叫电梯到这层。  
项目体现：traceid 的 `Ensure`、Outbox handler 用 UPSERT、payload 带 dedup_key。  
**为什么需要**：分布式系统中网络抖动、超时重试、Worker 重启都可能导致同一操作执行多次。幂等设计让"多次执行"变安全，否则你需要复杂的"防重"逻辑。这是分布式系统能稳定运行的基石。

## 5.7 Outbox Pattern（事务出箱模式）
业务库写入 + "待发事件"写入**在同一个数据库事务**里。  
解决"业务写了但消息没发"的丢失问题——保证事件不丢。
```sql
BEGIN;
INSERT INTO stories(...);
INSERT INTO outbox_events(event_type, payload, ...);
COMMIT;
```
**类比**：餐厅订单纸——"客人点了什么"和"送到几号桌"写在同一张纸上，绝不会做了菜不知道送哪儿。  
我们的核心卖点"有记忆的 AI"靠它兜底。  
**为什么需要**：传统做法是"写完业务库再发消息到队列"——两步之间崩溃就消息丢了。我们的核心卖点是"有记忆的 AI"，记忆更新如果丢了产品价值就破坏了。Outbox 用数据库事务把业务和事件绑在一起，崩溃也不丢。

## 5.8 优雅关停（Graceful Shutdown）
进程退出前先完成在途请求，再关闭。  
- ❌ `kill -9` 强杀：用户看到"故事生成到一半失败"
- ✅ `kill -TERM`（默认）：进程接到信号 → 拒绝新请求 → 等在途完成 → 退出  
项目体现：Task 18 main.go 监听 SIGTERM。  
**为什么需要**：发版时进程要重启——如果不优雅关停，正在生成故事的用户会突然失败。优雅关停给在途请求 30 秒"收尾时间"，用户感知不到部署过程。

## 5.9 业务错误码与 HTTP 状态码解耦（AppError 模式）
service 层抛错时怎么告诉 api 层"返 400 还是 500"？三种做法：
- ❌ service 直接调 `c.JSON(400, ...)` —— 业务依赖 HTTP 框架，污染分层
- ❌ 返回字符串错误 —— api 层凭字符串猜状态码
- ✅ 定义 `AppError`：含业务 Code（枚举） + UserMsg + `HTTPStatus()` 方法

```go
return errors.New(errors.CodeNotFound, "child_not_found", "未找到孩子档案")
```
api 层 `errors.As` 解出来一查就知道返 404。  
**为什么需要**：让 service 层完全不依赖 HTTP 框架——单元测试不用起 Gin，将来加 CLI/cron 入口也复用同一段业务。这是 [5.2 依赖倒置](#52-依赖倒置dip) 的具体落地。  
项目体现：`internal/pkg/errors` 包定义 7 种 Code 自动映射到 7 种 HTTP 状态。

## 5.10 同步 vs 异步
- **同步**：调用方等到被调方做完才返回（"做完再走"）
- **异步**：调用方下单后立刻返回，被调方事后慢慢做（"下单就走"）

**类比**：取快递——同步=柜员现场翻库 5 分钟；异步="放心今晚送你家"。
**为什么需要异步**：避免用户等"用户不必等"的事。爱宝故事生成中，"调 LLM"用户必须等（同步），"更新 MEMORY"用户不必等（异步）。把不必等的甩出去，体验最优。

## 5.11 消息队列（Message Queue）
异步任务的"待办清单"——一端放任务，另一端取任务做。  
**类比**：餐厅订单板，服务员贴单（生产者）+ 厨师撕单做菜（消费者）。生产消费解耦。  
主流：Kafka（超大流量）/ RabbitMQ（通用）/ Redis Stream（轻量）/ **PG Outbox**（最简单可靠）。  
**为什么 MVP 用 PG**：每天数百任务，PG 表够用；YAGNI，等真到 10 万+/日 再换 Kafka。

## 5.12 重试 + 指数退避（Retry with Exponential Backoff）
任务失败不立刻重试，等的时间越来越长（2s → 4s → 8s → ...）。  
**为什么需要**：失败常因"短暂限流/网络抖动"。立刻重试还是错；等一会儿再试可能成。指数让"系统恢复时间"和"重试间隔"匹配。  
**最大重试次数**：超过 N 次（如 5）入死信队列（DLQ），人工排查。

## 5.13 死信队列（Dead Letter Queue, DLQ）
重试 N 次仍失败的任务，**不再继续重试**，而是搬到一个"专门存放失败任务"的地方等人工处理——这就是死信队列。

**为什么不能无限重试**：
1. **资源浪费**——一个永远会失败的任务（比如代码 bug、数据格式错），无限重试只是反复消耗 CPU + 占用 Worker 槽位
2. **掩盖问题**——失败任务一直 pending 看着像"还在处理"，告警出不来；进 DLQ 才能触发"为什么有 5 个任务进 DLQ 了"的人工关注
3. **健康任务被堵**——Worker 反复处理同一个失败任务时，新进来的健康任务排在后面等

**怎么决定进 DLQ**：项目策略 = 重试 5 次仍失败 → status 改为 `dead`。**指数退避乘以 5 次** ≈ 2+4+8+16+32 = 62 秒，覆盖大部分"短暂故障"窗口；过了这个窗口还失败基本不是临时问题。

**DLQ 之后**：监控告警（每 N 个进 DLQ 触发一次） → 人工查看 last_error → 修代码 / 修数据 → 手动重置回 pending 让 Worker 重跑（或直接弃）。

**项目体现**：`outbox_events` 表的 `status` 字段有 `pending / processing / done / dead`；`OutboxRepo.MarkFailed` 在 `attempts >= maxAttempts` 时把 status 设为 `dead`，不再被 `FetchPending` 拉到。

**类比**：邮局送不出去的信。前几次"地址不全/收件人不在"会重试投递；超过 N 次进**死信处理中心**（real-life 的 DLQ）——邮政员人工开拆找原因，而不是无限往那个错地址送。

## 5.14 软件分层与依赖方向
```
api → service → repository + gateway → pkg
```
高层依赖低层，低层不依赖高层。Go 编译器**自动拦截循环依赖**——强制你想清楚分层。  
**为什么需要**：循环依赖（A 依赖 B、B 又依赖 A）是设计混乱的信号——意味着两层职责没分清。Go 编译器直接拒绝循环依赖，强制开发者**先想清楚谁依赖谁**——这是分层架构能落地的硬保证。

## 5.15 异步降低感知延迟（Perceived Latency）

用户体感的"快"不一定等于服务器的快——把"用户不必专注等待的事"甩到后台是 UX 设计的核心手段。**关键公式**：
```
感知延迟 = 总延迟 × 用户专注度
```
专注度为 0（用户已经在做别的事）时，再长的后台耗时用户也感觉不到。

**项目体现**（Plan 5 实测数据）：
- LLM 文本生成：16 秒（用户必须等——拿到文本才有内容可读）
- TTS 合成 + COS 上传：6 秒（用户不必等——读文本时音频已在做）

**两条路径对比**：
| 路径 | 用户开始读文本的时间 | 拿到音频的时间 | 感知体验 |
|---|---|---|---|
| 同步（等所有完成再返回） | 22s 后 | 22s 后 | 呆等 22s 一片空白 |
| 异步（POST 立即返回文本） | 16s 后 | 22s 后（但用户在读） | "拿到文本就开始"，音频"读完第一段就好" |

**绝对延迟**只多了 6 秒，**感知延迟**接近零——这就是异步的胜利。

**为什么需要这条**：很多团队优化 latency 只盯着"如何把 22s 降到 18s"——技术难度高、收益线性。换个思路把"用户不必等的"剥到后台，**绝对 latency 完全不变**也能让用户觉得快了一倍。这是工程师容易忽略、UX 设计师天然会想到的视角。

**类比**：餐厅上菜——好餐厅会先上开胃菜和茶水，让你"已经在吃"，主菜还在做但你已经不焦躁。差餐厅让你干坐 30 分钟才一次性端上来。客观时间一样，体感完全不同。

**判定哪些工作可以异步的口诀**：用户**拿到当前响应后会立刻消费**的，就是同步必须等的；用户**消费完当前响应才会用到**的，就可以异步。爱宝故事：文本=立刻读=同步必须等；音频=读完才会想"哎我要听版"=可以异步。

**项目体现细节**：`tts_synthesis` 写进 outbox 表（事务保证不丢），Worker 后台消费，audio_status 三态机供 client 轮询。这是 [5.7 Outbox Pattern](#57-outbox-pattern事务出箱模式) 与异步思想的具体结合。


## 5.16 Outbox 事件 payload 不可后改

Outbox Pattern 的事件一旦 INSERT 进 `outbox_events` 表，**Go 内存里那个 `*OutboxEvent` 对象的修改不会同步到 DB**。常见陷阱是"先 INSERT 占位、后填补真值"：业务代码在事务结束、拿到自增主键后想 patch payload 里的 id 字段——但 patch 只改了内存。Worker 拉到的 payload 永远是事务结束时的"占位"值。

**正确做法**（三选一）：
1. 用 outbox row 的 `aggregate_id` 字段持有那些"事务后才知道"的 id —— `CreateWithOutbox` 内部填，跨进程可见
2. Handler 进来时**再 query 一次**最新 entity（接受多一次 DB 往返）
3. 业务先写 entity 拿 id、再单独 INSERT outbox（**牺牲事务原子性**，不推荐）

**为什么需要**：分布式系统里"内存修改不等于持久化"是基本功，但**事务边界**让事情变隐晦——业务代码看到事务返回的 `entity.ID`，自然想"顺手 patch 一下"payload，殊不知事件已经写出去了。这是 Plan 4 埋下、Plan 6 暴露的真实 bug 模板。

**项目体现**：Plan 6 commit `4b522f3` — memory_update handler 改成"优先 `e.AggregateID`，payload.story_id 仅做兜底"。Plan 5 tts_synthesis 早就这么做了。

**类比**：邮件——投出后改本地草稿不影响收件人看到的内容。要更正只能再发一封。
