# LLM 工程

> Plan 4 起的 LLM 接入相关概念。本主题独立于"软件设计"和"安全合规"，因为 LLM 工程有它自己的范式和成本模型。

## 11.1 Token
LLM 处理文本的最小单位。**1 token ≈ 1.5 个汉字 ≈ 0.7 个英文单词**。  
**为什么这个单位重要**：
1. **LLM 按 token 计费**——比如豆包 Pro 约 0.8 元 / 100 万输入 + 2 元 / 100 万输出
2. **LLM 有"上下文上限"**——如 32K 表示一次最多 32000 token
3. 估算成本和拼 prompt 时都要算 token

**估算公式（中文）**：内容字数 × 0.67 ≈ token 数。  
项目体现：单次故事生成 ≈ 1500 token ≈ 0.003 元；100 元 / 天预算够 30000+ 次。

## 11.2 System Prompt vs User Prompt
分别承担"你是谁、规则是什么"和"用户具体问什么"两个职责：
```
[System] 你是爱宝，一只熊猫小机器人。永远不能...（长且固定）
[User] 讲个奥特曼睡前故事（短且变）
[Assistant] 好的，从前有个叫小宇的小朋友...
```
**为什么需要分开**：
1. LLM 训练时被教"system 优先级最高"——把不能违反的规则放 system
2. 缓存层面 system 通常一致，可以走"前缀缓存"省钱（豆包支持）
3. 责任边界清晰

项目体现：Plan 3 的 prompt.Builder 拼 System Prompt（含 8 条强约束）；用户的"讲个奥特曼故事"是 User Prompt。

## 11.3 Temperature（创意度参数）
控制 LLM 输出"放飞程度"的旋钮，**0 = 严肃保守，1 = 活泼有创意**（多数 API 上限 1 或 2）。  
**为什么需要**：同一个 prompt，温度低则每次输出几乎一样（适合分类、提取信息）；温度高则每次都有惊喜（适合创意写作）。  
项目策略：故事生成用 **0.8**——需要创意但要受控；意图分类用 **0**——要稳定可重复。

## 11.4 流式输出 vs 整体返回
- **流式（streaming）**：LLM 边生成边一字一字推回客户端（SSE 长连接）
- **整体返回**：全部生成完一次性返回（普通 HTTP 响应）

**为什么流式体验好**：用户看着字一个个蹦出来，等待感弱；体感"快"很多。  
**为什么 MVP 选整体返回**：代码简单，加 loading 动画即可。流式涉及 SSE / 客户端状态机，工程量大。  
项目体现：MVP 整体返回；v2 可升级流式。

## 11.5 Fallback（降级模板）
调 LLM 失败时返回**预先写好的安全故事**，让用户感知不到挂了。  
**项目策略**：
- 试 1 次失败 → 重试 1 次（可能是限流）
- 仍失败 → 用 fallback（一组预生成的"通用爱宝故事"）
- 主角名替换为孩子昵称

**为什么需要**：豆包偶尔抽风（限流/超时/服务异常）。fallback 让 P99 体验依然 OK，比"返回 500 让用户重新点"好得多。  
**类比**：餐厅厨师烧菜失败 → 端出预制菜 →客人吃饱了；总比"今天本店歇业"强。

## 11.6 上下文窗口（Context Window）
LLM 一次能处理的最大 token 数。豆包 Pro 32K = 一次 32000 token，约 21000 汉字。  
**为什么有上限**：transformer 架构的 attention 计算复杂度 O(N²)，更长上下文 = 更慢更贵。  
**MVP 不会触及**：System Prompt 1000 字 + User Prompt 100 字 + 故事输出 1500 字 ≈ 1.7K tokens，离 32K 还远。  
**何时会触及**：未来"连续剧"模式带 5 集历史上下文 + MEMORY 摘要，可能逼近 8K。再深需要升 128K/256K 模型。

## 11.7 Prompt 工程的"前缀缓存"省钱技巧
某些 LLM 提供商（包括豆包）支持"如果 prompt 前缀和上次一样，前缀部分按缓存价计费"。  
**项目用法**：System Prompt 是固定的（只随 prompt_version 改变），把它放在最前面 → 多数请求享受缓存价 → 实际成本可降 30-50%。  
**注意**：MEMORY 摘要、孩子档案这些**变动频繁**的内容放后面。变量少的内容前置，变量多的内容后置。

## 11.8 火山引擎 Endpoint ID 系统
豆包通过 OpenAI 兼容协议暴露，但 `model` 字段**不接受** `doubao-1.5-pro-32k` 这样的模型名称——必须先在火山控制台创建"推理接入点"绑某个具体模型，得到 `ep-m-XXXXX` 形式的 endpoint ID，**这个 ID 才是 API 调用时填的 `model` 字段**。

**为什么这样设计**：让 Volcengine 用户能为同一模型创建多个 endpoint（不同付费档位、不同地域、不同限流策略），每个 endpoint 是计费/路由/限流的最小单位——比"模型名称"更精细可控。

**项目体现**：`config.dev.yaml` 的 `story_model` 和 `intent_model` 字段都填 `ep-m-...` 格式（不是模型名）。新接入新模型时记得走"创建 endpoint → 拷 ep-m- ID → 写进配置"这个完整流程。

**踩坑提醒**：还要确认 endpoint 绑的模型与用法匹配——比如 Seedream（文生图）endpoint 不能用于纯文本分类调用，会 404。

## 11.9 fail-open vs fail-closed
当依赖外部服务失败时（LLM 超时、Redis 挂、数据库不可达等），系统怎么响应？两种策略：
- **fail-open**：放行用户请求 + 记日志告警 —— 适合"提示性"检查
- **fail-closed**：拒绝用户请求 + 返错 —— 适合"安全性"硬要求

**为什么需要显式声明**：复杂系统不可避免有依赖故障。如果不提前定义"挂了往哪边倒"，故障时行为就是随机的——有些请求过、有些挂，根本无法定位。所以必须**为每个依赖**显式定义 fail 方向。

**项目策略**：
- 意图分类 LLM 失败 → **fail-open** 到 IntentSafe（提示性检查，挂了不应连累用户）
- 红线词匹配 → **fail-closed**（硬安全要求，匹配过程不能跳过）
- 预算检查 Redis 出错 → **fail-open**（Redis 挂不应连累用户）
- 预算耗尽 → **fail-closed**（防止破产，超额必须拒）

类比：机场安检（fail-closed，宁错杀不放过）vs 餐厅 wifi（fail-open，登录系统挂了仍上菜）。


## 11.10 软提示 vs 硬提示的取舍

往 system prompt 注入"附加上下文"有两种风格：
- **软提示**：把信息以"事实段"放进 prompt，例如 "最近的故事记忆：小宇救了珊瑚"，让 LLM **自然回调**。优雅但效果不稳定。
- **硬提示**：明确指令 "请在新故事中至少借用一次海洋元素"。可控但容易显得刻意。

**实测**（Plan 6 smoke）：仅放"软提示"+ 说"可以自然回调"，LLM 在 user prompt 主导（"森林探险"）的情况下完全忽略 memory context，3 个候选关键词命中 0。

**何时选软提示**：
- LLM 已对场景非常熟悉（不需要被告知"怎么做"）
- 用户体验要"自然"超过"准确"（聊天伴侣 > 工具）
- 上下文足够独特（一次出现的人物/地名 LLM 一定会用）

**何时选硬提示**：
- 业务要求"必须出现 X"
- LLM 在 user prompt 主导时倾向忽略上下文
- 可以接受 LLM 输出"显得 mechanic"

**项目策略**：Plan 6 初版选软提示验证管线通畅；后续 Plan 6b 准备升级——把 memory 段从 system prompt 末尾抬到 "## 上下文" 单独章节，措辞改成 "请尝试在新故事中借用之前出现过的角色/场景，让孩子有'熟悉的延续感'"。

**为什么需要**：prompt 工程不是"写了就对"——同样的信息放不同位置、用不同措辞，LLM 行为差异巨大。新人容易写完"软提示"就以为完事，实测后才发现 LLM 没听。**先验证、后优化、再上线**是 prompt 工程的标准节奏。

**Plan 6b 实测进展**：Plan 6 时纯软提示零回调；Plan 6b 同时改"位置（末尾→IDENTITY 之后）"+"措辞（可以自然回调→请尝试借用）"+ memory 内容质量保持，**Story 23 真回调了 Story 22 的小精灵**。结论修正：软/硬提示不是二元，而是"位置×措辞×内容浓度"三维调节。**经验法则**：
- 位置：靠 system prompt 前部 ≈ 5 个权重 vs 末尾 ≈ 1 权重
- 措辞：祈使句（"请尝试"）≈ 3 权重 vs 陈述句（"可以"）≈ 1 权重
- 内容浓度：30 字精炼摘要 ≈ 2 权重 vs 200 字长文 ≈ 1 权重（长文会被 LLM 当背景）

三个维度任一调到中档以上，LLM 大概率响应；只调一个可能没效果。这是 Plan 6 → Plan 6b 单变量实验得出的实践。


## 11.11 LLM "顺势调用"模式：用低成本调用承担辅助任务

主调用做主任务时，**额外调一次低成本模型** 做"辅助产物"——这就是顺势调用模式。本项目落地两次：
- **Plan 6 memory.Summarizer**：故事生成后顺势调 doubao-lite 把全文压成 30 字 summary，存 memories.payload
- **Plan 8 chapter_hook.Extract**：故事生成后顺势调 doubao-lite 写 20 字下集预告，存 storylines.next_episode_hint

### 经济性

- 主调用：1500+ token 输入 + 数百 token 输出 ≈ 0.01-0.05 元（doubao-pro）
- 辅助调用：< 300 token 输入 + < 100 token 输出 ≈ < 0.005 元（doubao-lite 还便宜 5-10 倍）
- **辅助产物成本占比 ~10%，提升的"下次主调用上下文质量"是值得的**

### 典型模式

```go
// 1. 主调用产出主要内容
storyText := mainLLM.Generate(ctx, ...)   // 0.03 元，doubao-pro

// 2. 顺势抽辅助产物（fail-open，独立的 client + 便宜 model）
hint := hookLLM.Extract(ctx, storyText)   // 0.003 元，doubao-lite
// hookLLM 内部 LLM 错时返 "" 不抛错，主流程不受影响

// 3. 持久化（辅助产物可有可无）
storyRepo.Save(ctx, story)
storylineRepo.IncrementEpisode(ctx, slID, hint)  // hint="" 时也写入，下次 prompt 跳过空 hint 分支
```

### 辅助调用应当具备的属性

- **短输出**（< 100 token）—— 避免主流程被慢辅助调用拖累
- **用便宜模型**（doubao-lite 而非 doubao-pro）
- **fail-open**（空字符串而非 error 上抛）—— 辅助产物缺失不应失败主流程
- **同步紧跟主调用**（不进 outbox，反正辅助产物可有可无；进 outbox 反而增加 latent bug 面）
- **结构化产物**（短字符串 + 存 jsonb 或单列），方便下次主调用拼进 prompt

### 为什么需要

LLM 应用工程师经常陷入"一次 LLM 调用做完所有事"的局部最优——结果 prompt 越写越长、输出格式越来越复杂、主调用越来越脆。**顺势调用模式**把"主内容生产"和"辅助元数据提取"分离：
- 主调用专注内容质量
- 辅助调用专注结构化提取
- 两者用不同模型 + 不同 prompt，**各自能独立调优**

**类比**：餐厅厨师做完主菜后，**顺势**问"要不要给桌号 5 留个'喜欢辣口'的标签？"——三秒钟的事，下次来就能预设辣口。如果让主厨在做菜过程中同时记录顾客偏好，菜会做不好。

### 反模式：辅助调用进 outbox 异步

诱惑：辅助调用也异步、塞进 outbox、Worker 后台处理。**不推荐**——辅助产物的价值是"喂回下次主调用作为上下文"，如果异步处理，**它和主写入不在同一个时间点完成**，下一次主调用时可能还没出来，导致上下文丢失。**同步紧跟**是这个模式的灵魂。

## 11.12 字数硬约束：LLM 没有原生字数感，必须靠 prompt + 校验双重保险

🎓 让 LLM 严格输出"约 1500 字"很难——它没有字数感，凭语感停笔。

**爱宝项目实测**（5 分钟挡目标 1440–1760 字）：
- 输出 600 字（差 60%）：~30% 概率
- 输出 1500 字（命中）：~50%
- 输出 2000 字（超写）：~20%

**单纯改 prompt 提高命中率有限**。今天试了三档力度：

| Prompt 表达 | 命中率（5min 挡） |
|---|---|
| "目标 5 分钟，约 600 字" | 极低 |
| "目标 5 分钟，约 1500 字" | ~30% |
| "**硬约束**：必须输出 1400–1600 个汉字（不含标点和音效标记）。如果情节展开不够，请增加细节描写、对话、孩子的内心活动" | ~50% |

**真正解决方案**：**prompt + 后置长度校验 + 反馈式重写**

```go
text := llm.Generate(systemPrompt, userPrompt)
got := CountCJKRunes(text)   // 数纯汉字，跳过标点 / 空格 / [音效:xx]
for retryNo := 1; retryNo <= 2 && got < expectedMin*0.7; retryNo++ {
    retryUser := fmt.Sprintf(
        "%s\n\n【重要】上次只写了 %d 字，必须超过 %d 字。请增加细节描写、对话、内心活动来扩展。",
        userPrompt, got, expectedMin,
    )
    resp := llm.Generate(systemPrompt, retryUser)
    if newGot := CountCJKRunes(resp); newGot > got {
        text = resp; got = newGot
    }
}
```

**关键设计点**：
1. **反馈具体数字**：告诉 LLM "上次写了 X 字"远比 "请写长一点"有效
2. **告诉它怎么扩展**：不是"多写点"，而是"用细节描写 / 对话 / 内心活动 / 感官细节"
3. **设上限**：最多重试 2 次，避免无限循环烧钱
4. **只采纳更长版本**：第二次比第一次短就保留原版（防止恶化）

**为什么需要**：LLM 用户的常见痛点是输出长度不可控。仅靠 prompt 是"祈祷模型听话"，**长度校验 + 反馈式重试**把"长度"变成可工程化的指标。

**何时引入**：Plan 9c Task 1 / commits `7d84c81` + `62b8613`。

## 11.13 SceneSeed 模式：用随机种子注入熵打破 LLM 同质化

🎓 LLM 在低多样性输入下会**严重同质化**——同样的 prompt、同样的 memory、同样的 hook → 一模一样的开头。

**爱宝实测**：连续 6 个故事所有开头都是「夜晚，月光洒在房间里，爱宝蹦蹦跳跳来到小宇床边」。原因：
- system prompt 模板固定
- memory 注入的摘要每次都是同一场景
- temperature 0.8 不足以拉开开头

**单纯调 temperature 不够**：拉到 1.0+ 会让故事散乱、违反约束。

**SceneSeed 模式**：在 prompt 里**手动注入一个随机种子**，强行打破开头同质性。

```go
var sceneSeeds = []string{
    "清晨的厨房里飘着早餐的香味",
    "夏天的海边沙滩，浪花轻轻拍过来",
    "动物园里大熊猫吃竹子的展区",
    // ... 80 个，分 9 大类
}
// 每次随机抽一个，注入：【本次场景灵感】<seed>
```

**配合反套路明文**：
```
【避免套路（重要）】
- 不要每个故事都以"夜晚 / 房间 / 月光 / 床边"开场。
- 不要用"今天咱们去 XX 吧"启动剧情。
```

**概率账**：
- 池子 20 个：连续 6 个故事重复率 ~0.5%
- 池子 80 个：连续 6 个 ~0.03%，连续 20 个 < 5%

**为什么需要**：**便宜熵**——比训练 LoRA、微调模型简单一万倍，且不需要任何模型侧改动。"输入熵决定输出熵"的最简单实现。

**何时引入**：Plan 9c Task 3 / commits `ec33a63` + `a6e2cbf`。

## 11.14 嵌套硬约束在产品演进中的"互相绞杀"

🎓 多个硬约束叠加时，每条单独看都合理，组合起来会让产品落到"最差路径"。

**爱宝项目活案例**：
- **Plan 8** 加的约束：续集故事必须命中至少 1 个上一集元素，否则 PostCheck fail
- **Plan 9c** 加的约束：故事字数必须 ≥ 期望 × 70%，否则触发重写

**结果**：用户生成 3 min 续集 →
1. LLM 写了完整故事，但用了新场景"星星城堡"（上一集里没有）
2. Plan 8 硬约束 fail → 走 fallback
3. 3 min fallback 模板只有 150 字 → **45 秒音频**

用户体验完全崩盘。

**修法**：把 Plan 8 续集约束从 **hard-fail** 改为 **warn-only**：宁可故事不完全连贯，也别给用户 45 秒罐头。其他 PostCheck reason（安全 / 主角错位 / 恐惧词）仍 hard-fail。

**复盘原则**：
1. **每条硬约束都要问"失败时降级到什么"**：如果降级体验比"放过"更差，改成 warn-only
2. **新约束跟旧约束的交叉效应**是产品演进里最易爆雷的地方
3. **业务硬约束**（连续性 / 字数）和 **安全硬约束**（红线词 / 暴力 / 政治）要分级处理，不能一刀切

**为什么需要**：加约束"看起来在改善质量"，但叠加效应可能让用户撞到比"不加约束"更糟的状态。**约束系统设计要考虑组合爆炸**。

**何时引入**：Plan 9c Task 4 / commit `96a86f4`。
